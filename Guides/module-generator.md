# Module Generator

If you're starting a new FreePBX module, this guide walks you through the scaffold you need and how to wire it into a running system. Sangoma ships a `freepbxgenerator.phar` tool for bootstrapping modules; the [helloworld](https://github.com/FreePBX/helloworld) module is the best reference for what a modern BMO layout looks like in practice. The `moduleclass.stub.php` gist adds AJAX and grid patterns on top of that baseline.

Older docs (including [Module Lifecycle](../Core Concepts/module-lifecycle.md)) describe generator output that still includes `install.php` and `uninstall.php`. For FreePBX 12 and later, put install and uninstall logic in your BMO class instead — `helloworld` has no standalone `install.php`.

## Recommended Scaffold (Modern BMO)

This layout matches `helloworld` on `release/15.0`:

```
{rawname}/
├── {Rawname}.class.php       # BMO primary class (namespace FreePBX\modules)
├── module.xml                # Manifest (metadata, depends, database, menuitems)
├── page.{rawname}.php        # Thin page entry → $freepbx->{Rawname}->showPage()
├── views/                    # HTML templates (no business logic)
│   ├── default.php
│   ├── form.php
│   └── grid.php
├── assets/
│   ├── js/{rawname}.js
│   └── css/                  # optional
├── Backup.php                # optional — backup engine integration (15+)
├── Restore.php               # optional — restore engine integration (15+)
├── Console/
│   └── {Rawname}.class.php   # optional — fwconsole commands
├── Api/
│   ├── Rest/                 # optional — REST API providers
│   └── Gql/                  # optional — GraphQL resolvers
├── ucp/                      # optional — User Control Panel module
│   ├── {Rawname}.class.php
│   ├── assets/
│   └── views/
├── .github/workflows/        # optional — CI signing (see [Contribution Workflow](contribution-workflow.md))
│   ├── signrelease.yml
│   ├── sign.sh
│   └── exclude.txt
├── LICENSE.md
└── README.md
```

## Required Files (Minimum Viable Module)

At minimum you need a manifest, a BMO class, and (if you have a GUI) a page entry point.

### `module.xml`

```xml
<module>
    <rawname>myapp</rawname>
    <repo>unsupported</repo>
    <name>My Application</name>
    <version>1.0.0</version>
    <category>Applications</category>
    <menuitems>
        <myapp>My Application</myapp>
    </menuitems>
    <depends>
        <version>15.0</version>
    </depends>
</module>
```

See [module.xml Manifest Structure](../Core Concepts/module-xml-manifest.md) for the full element reference.

### `{Rawname}.class.php`

From `moduleclass.stub.php` — the minimum BMO contract:

```php
namespace FreePBX\modules;

class Myapp implements BMO
{
    public function __construct(FreePBX $freepbx) { ... }
    public function install() {}
    public function uninstall() {}
    public function doConfigPageInit($page) {}
}
```

These methods are required even when empty. `install()` replaces legacy `install.php`; `uninstall()` replaces `uninstall.php`.

### `page.{rawname}.php`

```php
<?php
echo FreePBX::create()->Myapp->showPage();
```

Keep the page file thin — delegate all logic to the BMO class.

## Stub-Provided Optional Methods

`moduleclass.stub.php` goes beyond the minimum contract with patterns you can copy and customize:

| Method | Purpose |
|--------|---------|
| `getActionBar($request)` | Submit/Delete/Reset buttons per display |
| `ajaxRequest($req, &$setting)` | Declare handled AJAX commands |
| `ajaxHandler()` | Process AJAX (e.g., Bootstrap Table `getJSON`) |
| `getRightNav($request)` | Right-side navigation HTML |

Grab these from the gists rather than writing them from scratch.

## Legacy Generator Output

If `freepbxgenerator.phar` still emits the older layout, you'll see something like this:

```
{rawname}/
├── install.php          ← legacy; BMO install() preferred
├── uninstall.php        ← legacy; BMO uninstall() preferred
└── ucp/ (optional)
```

Move any logic from `install.php` and `uninstall.php` into your BMO class before targeting FreePBX 15+.

## Post-Scaffold Steps

### Local development install

```bash
# Symlink or copy module into framework modules directory
ln -s /path/to/myapp /var/www/html/admin/modules/myapp

# Install and enable
fwconsole ma install myapp
fwconsole ma enable myapp
fwconsole reload
```

The generator phar can symlink into the framework module directory for you — check the Sangoma wiki for current flags and behavior.

### Database tables

Declare schema in the `module.xml` `<database>` section. The framework creates tables on `fwconsole ma install`. See [module.xml Database Section](../Core Concepts/module-xml-database-section.md).

### GUI wiring

1. Add a `<menuitems>` entry in `module.xml`
2. Create `page.{rawname}.php` and `views/`
3. Implement `showPage()` and `doConfigPageInit()` in your BMO class
4. See [GUI Page Routing, showPage, and Views](../Core Concepts/gui-pages-and-views.md)

## Naming During Generation

Apply conventions from `../Advanced Topics/naming-conventions.md` when you pick a rawname:

| Input | Output |
|-------|--------|
| rawname `myapp` | Class `Myapp`, file `Myapp.class.php` |
| rawname `my-module` | Class `Mydash` (hyphen → dash) |
| GUI display | Matches `<menuitems>` key = rawname |
| Page file | `page.myapp.php` |

## Things to watch for

- `freepbxgenerator.phar` is a Sangoma tool — see their wiki for phar-specific flags and auto-install behavior
- Modern modules should not rely on `install.php`/`uninstall.php`; use BMO methods instead
- Generator output may omit `Backup.php`/`Restore.php`, `Api/`, and `Console/` — add those manually when you need them
- Set `<repo>unsupported</repo>` for community modules
- Your class must implement the `BMO` interface and live in the `FreePBX\modules` namespace
- Do not use a closing `?>` in PHP files (see [Code Style Conventions](../Core Concepts/code-style-conventions.md))
