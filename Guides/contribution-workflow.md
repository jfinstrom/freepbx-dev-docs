# Contribution Workflow

This guide covers the git branching, release automation, and ticket-reference patterns you'll see in FreePBX module repos. It's based on `helloworld/.github/workflows/`, `helloworld/README.md`, `moduleAdminFunctions.php`, and framework source comments.

Sangoma's internal review and merge process isn't documented here — this focuses on what you can observe and replicate in your own module repository.

## Repository Branch Conventions

The `helloworld` reference module uses `release/15.0` to track FreePBX 15.x.

CI in `signrelease.yml` runs on push to these branch prefixes:

| Branch pattern | Purpose |
|----------------|---------|
| `release/*` | Versioned release builds (e.g., `release/15.0`) |
| `bugfix/*` | Bug-fix branches |
| `feature/*` | Feature development branches |

All three prefixes trigger the same sign-and-release workflow.

## Automated Release Pipeline

On push to a qualifying branch, GitHub Actions (`signrelease.yml`) runs:

```
1. Checkout repository
2. Install gnupg + xmlstarlet
3. Import GPG_SIGNING_KEY secret
4. Extract <version> and <rawname> from module.xml
5. Run sign.sh → module.sig + {rawname}-{version}.tar.gz
6. Create GitHub Release tag v{version} with tarball attached
```

### sign.sh responsibilities

| Step | Output |
|------|--------|
| Parse `module.xml` for version/rawname | Tarball name |
| SHA-256 hash all module files | `module.sig` `[hashes]` section |
| GPG clearsign signature file | `module.sig` |
| `tar -czf` with exclude list | `{rawname}-{version}.tar.gz` |

Release tags use the format `v{version}` — for example, `v15.0.1` from helloworld's `module.xml`.

## Version and Manifest Discipline

Before you push a release branch:

1. Bump `<version>` in `module.xml`
2. Update `<changelog>` with release notes
3. Ensure `<rawname>` matches the directory name and BMO class file prefix
4. Push to the appropriate branch (`release/*`, `bugfix/*`, or `feature/*`)
5. Let CI produce the signed tarball and GitHub release

## Ticket Reference Conventions

### Source code comments

Framework and module sources reference Jira tickets inline:

```php
// FREEPBX-7204 - Discard any cruft, errors, etc, that may have been
// http://issues.freepbx.org/browse/FREEPBX-9898
```

The pattern is `FREEPBX-{number}` in comments, often with a link to `http://issues.freepbx.org/browse/FREEPBX-{number}`.

Branch names with `FREEPBX-XXXXX` show up in some projects, but only `release/*`, `bugfix/*`, and `feature/*` are wired into the CI config we can see here.

### Changelog ticket linking

`moduleAdminFunctions.php::jira_replace_ticket()` auto-links ticket numbers in changelogs:

```php
// Matches 'FREEPBX-nnn', 'FPBXDISTRO-nnn' → Jira browse URL
function jira_replace_ticket($match) {
    $baseurl = 'http://issues.freepbx.org/browse/'.$match[1].'-';
    ...
}
```

Changelog entries commonly use the `*version*` prefix format:

```xml
<changelog>
    *15.0.1* FREEPBX-12345 Fix something
    *14.0.1* Code cleanup
</changelog>
```

Sangoma Atlassian tickets are supported too via `sangoma_atlassian_replace_ticket()`.

## Module Development Lifecycle

A typical flow looks like this:

```
feature/* or bugfix/*
    → development + local testing (fwconsole ma install)
    → merge to release/{major}.{minor}
    → bump module.xml version
    → CI signs + publishes GitHub release
    → users install via Module Admin or manual tarball upload
```

For unsupported or community modules, signing is optional. Unsigned modules trigger warnings when `SIGNATURECHECK` is enabled but still work.

## Practical Contribution Paths

### Community / third-party module

1. Fork or create a repo following the `helloworld` structure
2. Set `<repo>unsupported</repo>` in `module.xml`
3. Use a `release/*` branch with CI signing (optional), or distribute a tarball manually
4. Document in your README; link to the FreePBX wiki and gists for patterns

### Framework bug fix

1. Clone `FreePBX/framework` on the appropriate release branch
2. Reference `FREEPBX-{number}` in commit comments when applicable
3. Follow framework code style (`../Advanced Topics/code-style-conventions.md`)
4. Submit through Sangoma's established contribution channel

## Things to watch for

- GPG signing in CI requires a `GPG_SIGNING_KEY` GitHub secret — Sangoma's official keys aren't available to third parties
- `sign.sh` uses key ID `268C8DD0` for clearsign — that's Sangoma infrastructure
- The release workflow assumes a single-module repo at the root (not a monorepo)
- `release/15.0` in helloworld doesn't mean every module uses the same branch naming — check each repo
