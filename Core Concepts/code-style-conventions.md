# Code Style Conventions

FreePBX PHP code follows consistent formatting norms across the framework and modules. If you're contributing to core or building a new BMO module, matching these conventions keeps your code review smooth. The `helloworld` module and `freepbx-gists/moduleclass.stub.php` are good templates to start from.

## Vim Modeline (Framework Standard)

Most framework BMO files open with:

```php
<?php
// vim: set ai ts=4 sw=4 ft=php:
```

What each flag means:

| Flag | Meaning |
|------|---------|
| `ai` | Auto-indent |
| `ts=4` | Tab width displays as 4 spaces |
| `sw=4` | Shift width 4 for indent operations |
| `ft=php` | PHP filetype |

This is the documented Sangoma/FreePBX convention for PHP indentation settings.

## Indentation

Framework and BMO files use **tab characters** for indentation, not spaces:

```php
	public function install()
	{
		$fcc = new \featurecode('helloworld', 'helloworld');
	}
```

Match the surrounding file style when editing existing modules. New BMO files should follow the vim modeline + tab indent pattern.

## PHP Open Tag

```php
<?php
// No closing ?> at end of pure PHP files
```

The helloworld README warns:

> Avoid using `?>` followed by an empty newline, as PHP may encounter issues. Only use `?>` for inline PHP.

## Class Structure

Standard BMO module skeleton:

```php
<?php

namespace FreePBX\modules;

use BMO;
use FreePBX_Helpers;

class Helloworld extends FreePBX_Helpers implements BMO
{
	public $FreePBX = null;

	public function __construct($freepbx = null)
	{
		if ($freepbx == null) {
			throw new Exception("Not given a FreePBX Object");
		}
		$this->FreePBX = $freepbx;
		$this->Database = $freepbx->Database;
	}
}
```

Class structure conventions:

| Convention | Detail |
|------------|--------|
| `implements BMO` | Required for module class |
| `extends FreePBX_Helpers` | Modern pattern (helloworld) |
| `$this->FreePBX` | Store BMO reference |
| Braces | K&R style ã opening brace on same line for methods |

## Naming in Code

| Item | Style |
|------|-------|
| Methods | `camelCase` ã `doConfigPageInit`, `getActionBar` |
| Class | `PascalCase` ã `Helloworld` |
| Variables | `$camelCase` or `$snake_case` ã mixed in legacy code |
| Constants | `UPPER_SNAKE` ã `MODULE_STATUS_ENABLED` |
| Private/protected | No strict prefix; framework uses `private $` |

## Translation Wrapping

User-visible strings must be wrapped:

```php
_('Hello World')                           // GUI with active textdomain
modgettext::_('Hello World', 'helloworld') // Explicit module domain
```

See [i18n and Localization](../Advanced Topics/i18n-localization.md).

## Comments

Framework uses PHPDoc blocks on public methods:

```php
/**
 * Installer run on fwconsole ma install
 *
 * @return void
 */
public function install() { }
```

Avoid verbose comments on obvious code. Match the documentation level of the surrounding file.

## Legacy vs BMO Style

| Aspect | Legacy (`functions.inc.php`) | BMO |
|--------|------------------------------|-----|
| Structure | Procedural functions | Class methods |
| Indent | Mixed | Tabs + vim modeline |
| Prefix | `{mod}_function()` | Method names without prefix |

## Things to watch for

- Don't reformat unrelated code in drive-by changes
- Preserve the vim modeline in new framework-style PHP files
- Use tabs for indent in module PHP, matching helloworld/framework
- No closing `?>` on class files
- Wrap all user-facing strings with `_()` or `modgettext::_()`
